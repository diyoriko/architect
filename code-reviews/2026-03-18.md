## Cross-Project Patterns

**1. Stale test assertions confirming bugs**
Vedic Türkiye `types.test.ts` passes while asserting legacy (wrong) prices. This is a testing anti-pattern where green CI creates false confidence. When writing tests for business-critical constants (prices, model IDs), pin the *correct* value and let the test fail until the code is fixed.

**2. P0 items surviving multiple sprints**
Hunter's model ID fix (P0, Sprint 9) and SAMI's JSON.parse crash (critical, flagged 2026-03-17) are both multi-week open items. The review cycle is working (finding issues) but the fix pipeline is not closing them. Consider a hard rule: no new features merged while a P0/Critical is open.

**3. CI health correlation with shipping confidence**
Vedic Türkiye's CI has been red for 5 consecutive runs. Developers cannot trust a green badge because there isn't one. Lint errors are trivially fixable — the cost of leaving CI broken is higher than the 30 minutes to fix them.

**4. Silent external service failures**
GoatCounter (Portfolio) and the SAMI approval context (memory loss on restart) both fail silently. Users and operators see no error. Add health-check assertions or monitoring alerts for any service whose failure mode is "works but produces wrong results."
